# Пайплайн: транскрибация аудио с разметкой по спикерам

Этот документ — инструкция для нейросети (или человека), чтобы воспроизвести
рабочий пайплайн «аудио → текст с репликами по говорящим». Он написан по итогам
реального проекта, где сравнивались четыре инструмента, и объясняет не только
«как», но и «почему именно так» — включая ошибки, которых надо избежать.

---

## 1. Задача

**На входе:** аудиофайл с диалогом (любой формат: mp3, opus, wav, m4a...).
Обычно разговор двух людей (например, консультация), длительность до 1–2 часов.

**На выходе:** текст, разбитый на реплики, где у каждой реплики есть:
- таймкод начала и конца,
- имя говорящего («Спикер 1», «Спикер 2»),
- сам текст с пунктуацией.

Форматы вывода: `.txt` (реплики построчно), `.srt` (субтитры),
опционально `.md` (читабельный документ со статистикой).

---

## 2. Три компонента пайплайна

### Компонент 1: Распознавание речи (ASR)
Превращает звук в текст. Используется **GigaAM v3** от Сбера — на русском
он точнее Whisper. Модель `v3_e2e_rnnt` сразу ставит пунктуацию и заглавные
буквы (это важно — отдельный шаг пунктуации не нужен).

- Ставится ТОЛЬКО с GitHub: `pip install "gigaam[longform] @ git+https://github.com/salute-developers/GigaAM.git"`
  (версия на PyPI устаревшая, в ней нет моделей v3 — это частая ошибка).
- Длинные файлы обрабатываются методом `transcribe_longform()` — он сам режет
  аудио на куски по паузам (VAD) и возвращает сегменты по 15–20 секунд.
- **Критически важно:** вызывать с `word_timestamps=True` — тогда у каждого
  слова будет свой таймкод. Без этого качественное сведение невозможно (см. п. 3).
- Результат: список сегментов, у каждого поля `.text`, `.start`, `.end`
  и `.words` (список слов, у каждого свои `.text`, `.start`, `.end`).

### Компонент 2: Диаризация (кто когда говорил)
Составляет «карту голосов»: непрерывные интервалы времени с меткой говорящего.
Используется **pyannote/speaker-diarization-3.1**.

- Модели закрытые (gated): нужен токен Hugging Face, и на страницах
  `pyannote/segmentation-3.0` и `pyannote/speaker-diarization-3.1` надо один раз
  принять условия использования — иначе будет ошибка доступа.
- Если число говорящих известно точно — ОБЯЗАТЕЛЬНО передать `num_speakers=2`
  (или сколько есть): это заметно повышает точность. Есть также
  `min_speakers`/`max_speakers` для диапазона.
- Аудио лучше передавать готовым тензором `{"waveform": ..., "sample_rate": ...}`,
  а не путём к файлу — это обходит проблемы pyannote с чтением файлов.
- В pyannote 4.x результат обёрнут в объект: если у результата есть атрибут
  `.speaker_diarization` — брать разметку из него (в 3.x возвращается напрямую).
- Показ прогресса: `from pyannote.audio.pipelines.utils.hook import ProgressHook`,
  вызывать пайплайн внутри `with ProgressHook() as hook:` с `hook=hook` —
  будут видны этапы (сегментация → эмбеддинги → кластеризация).
- Результат: список «ходов» — троек (начало, конец, спикер).

### Компонент 3: Сведение (самое важное!)
Склейка текста с картой голосов: какому слову какой спикер.

**Это единственное место, где пайплайны отличаются, и именно оно определяет
качество результата.** ASR и pyannote у всех инструментов одинаковые.

---

## 3. Почему наивное сведение даёт кашу (главный урок)

Наивный способ: взять сегмент ASR (15–20 секунд) и приписать его целиком
одному спикеру — тому, кто говорил в середине сегмента, или тому, чьего
времени в сегменте больше. Так делают многие готовые инструменты
(GigaAMGUI, DialogScribe) — и дают плохой результат.

Причина: ASR режет речь по ПАУЗАМ, а не по смене говорящего. В живом диалоге
за 17 секунд успевают высказаться оба: вопрос одного + ответ второго попадают
в один сегмент, и весь сегмент подписывается одним именем. Короткие вопросы
и реплики «Да», «Ну да» систематически тонут в длинных ответах собеседника.

**Правильный способ (так работают Whisper XXL и этот пайплайн):** использовать
таймкоды отдельных СЛОВ. Слово приписывается тому спикеру, кто по карте
голосов говорил в момент его произнесения. Важно понимать: слово не
«распознаётся по тембру» отдельно — оно просто накладывается на карту голосов
по времени. Ошибки поэтому не рассыпаются по отдельным словам, а идут по
границам ходов, которые pyannote уже сгладил.

---

## 4. Алгоритм сведения (пошагово)

Вход: сегменты ASR (с пословными таймкодами) + ходы спикеров из pyannote.

**Шаг 1. Определить, «чистый» сегмент или «смешанный».**
Для каждого сегмента ASR посчитать, сколько секунд в нём принадлежит каждому
спикеру (по пересечению с ходами). Если у «второго» голоса меньше 15% времени
сегмента И меньше 1,5 секунды — сегмент чистый.

**Шаг 2. Чистый сегмент → целиком одному спикеру.**
Берём текст сегмента как есть (сохраняется родная пунктуация ASR) и приписываем
спикеру с максимальным пересечением. По словам не режем — незачем.

**Шаг 3. Смешанный сегмент → резать по словам.**
Каждое слово получает спикера по максимальному пересечению своего интервала
с ходами (если пересечения нет — ближайший по времени ход). Подряд идущие
слова одного спикера склеиваются в реплику.

**Шаг 4. Сглаживание «островков».**
Если получилась реплика короче ~1 секунды, зажатая между двумя репликами
ОДНОГО И ТОГО ЖЕ другого спикера, — переприписать её этому спикеру.
Это страховка от случайных одиночных слов, «выдранных» на границах.
Повторять проход, пока есть изменения.

**Шаг 5. Финальная склейка.**
Подряд идущие реплики одного спикера объединить в одну (текст через пробел,
время от начала первой до конца последней).

**Шаг 6. Имена спикеров.**
Метки pyannote (SPEAKER_00...) переименовать в «Спикер 1», «Спикер 2» —
в порядке первого появления в разговоре.

Параметры для подстройки (значения по умолчанию проверены на практике):
- `MIXED_SHARE = 0.15` и `MIXED_SECONDS = 1.5` — порог «смешанности» из шага 1.
  Меньше — агрессивнее резка по словам.
- `MIN_ISLAND = 1.0` — порог «островка» из шага 4. Если короткие «Да» одного
  спикера прилипают к чужим репликам — уменьшить до 0.5; если в результате
  появляется мусор из обрывков — увеличить до 1.5.

## 5. Подготовка аудио

Перед обработкой привести файл к виду, который ждут модели:
```
ffmpeg -y -i вход.любой -ac 1 -ar 16000 выход.wav
```
(моно, 16 кГц — стандарт для GigaAM и pyannote).

---

## 6. Порядок работы пайплайна целиком

1. Конвертация в WAV 16 кГц моно (ffmpeg).
2. Диаризация pyannote (с `num_speakers`, если известно) → карта ходов.
3. ASR GigaAM `transcribe_longform(wav, word_timestamps=True)` → сегменты со словами.
4. Сведение по алгоритму из п. 4 → список реплик (начало, конец, спикер, текст).
5. Запись результатов:
   - `.txt`: `[ЧЧ:ММ:СС - ЧЧ:ММ:СС] Спикер N: текст` построчно;
   - `.srt`: стандартные субтитры, спикер в начале строки текста;
   - `.md` (опционально): шапка со статистикой + диалог вида `**Имя:** текст`.
6. Статистика для шапки `.md`:
   - длительность файла;
   - доля времени речи каждого спикера (сумма длительностей его реплик / общая речь);
   - доля текста каждого спикера (сумма символов его реплик / все символы).

## 7. Известные грабли (проверено на собственном опыте)

- **gigaam с PyPI не знает моделей v3** — ставить только с GitHub (см. п. 2.1).
- **pyannote 3.1.1 несовместим с NumPy 2.x** (падает на `np.NaN`) — использовать
  pyannote >= 3.3 или патчить `np.NaN = np.nan` до импорта.
- **В Google Colab** после установки пакетов обязателен перезапуск сессии;
  предустановленные torchvision и onnxruntime конфликтуют с новыми пакетами —
  проще удалить оба (`pip uninstall -y torchvision onnxruntime`), пайплайну
  они не нужны.
- **pyannote 4.x** возвращает обёртку — проверять атрибут `.speaker_diarization`.
- **Поля сегментов gigaam** в свежих версиях — объекты с `.text/.start/.end`,
  в старых — словари с `'transcription'/'boundaries'`. Проверять оба варианта.
- **GPU:** модели требуют нормальной видеопамяти (от 4–6 ГБ). На слабых GPU
  (2 ГБ) диаризация работает часами — лучше CPU или облако (Colab T4 делает
  часовую запись за ~3 минуты). fp16 на старых GPU (архитектура Pascal и
  старше) МЕДЛЕННЕЕ, чем fp32 или CPU.
- **Похожие тембры голосов** — главный источник оставшихся ошибок диаризации.
  Подсказка `num_speakers` обязательна; спорные места правятся руками.
